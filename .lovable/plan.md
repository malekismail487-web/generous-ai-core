# Plan: Adaptive Intelligence v2.2 — Phases 1–5 SHIPPED + bonus pushes

Phase 5 (nightly anchor recalibration) + guess/slip dampening + info-gain CAT selector are now live on top of the v2.1 stack.

The current engine (subject + concept IRT with keyword inference) is a strong ~90% prototype. To reach production-grade adaptive accuracy, we layer in **7 targeted upgrades** that fix the highest-impact weaknesses: hard labels, brittle keyword matching, static difficulty, fragmented updates, no time decay, equal-weight updates, and no recalibration anchor.

Each change is independently shippable, ordered by impact.

---

## The 7 upgrades

### 1. Probabilistic concept assignment (+4–6%)

Replace single `conceptId` with a soft distribution: `[{concept, weight}, ...]` (top-2, weights sum to 1.0). Every concept update is scaled by its weight. Fixes mislabel damage and lets multi-skill questions feed multiple concept thetas proportionally.

```text
inferConceptId(text)  →  inferConceptDistribution(text, topK=2)
                          returns [{conceptId, weight}]
```

### 2. Embedding-based concept inference (+3–5%)

Keep keyword scoring as a fast fallback, but add an embeddings path for harder cases:

- Precompute concept-centroid vectors per subject (cached in a static JSON, generated by a one-time script using Lovable AI `google/gemini-embedding-001`).
- At inference time: if keyword score is weak (< threshold) but text is long enough, embed the question and pick top-2 concepts by cosine similarity.
- Hybrid score = `0.6 * cosineSim + 0.4 * normalisedKeywordScore`.
- Edge function `infer-concept` handles the embedding call (key stays server-side).

### 3. Dynamic difficulty self-healing (+3–7%)

Currently `question_bank.difficulty_b` only recalibrates at 20 attempts. Add a continuous nudge applied after each answer:

```
b_new = b_old + α * (observed_p - expected_p)   // α = 0.02, clamped ±3
```

Runs inline in the `ability-update` edge function so questions become better-calibrated as data accumulates, not in one cliff at n=20.

### 4. Hierarchical Bayesian coupling (+2–4% stability)

Concept theta should not drift arbitrarily far from subject theta. Add a soft pull during update:

```
concept_theta ← concept_theta + λ * (subject_theta - concept_theta)   // λ = 0.05
```

Applied **after** the Rasch update, not before. Stops noisy single-concept spikes from one bad day.

### 5. Temporal decay + reinforcement (+4–8%)

Add time-aware ability when reading (not writing) theta:

```
days_since = (now - last_graded_at) / 86400
effective_theta = theta * exp(-days_since / HALF_LIFE_DAYS)   // 60 days
effective_se    = se   + decay_uncertainty(days_since)
```

The stored `theta` is the raw value; the *effective* theta is what the prompt sees and what the next CAT question targets. Recent reinforcement (last 7 days, ≥3 answers) cancels decay.

### 6. Uncertainty-aware update gating (+2–5%)

Today every answer moves theta the same amount. Scale the Rasch update step `K` by signal quality:

```
K_effective = K_base * concept_weight * question_confidence * response_confidence
```

Where:

- `concept_weight` = the soft-assignment weight from #1.
- `question_confidence` = `min(1, times_seen / 20)` from `question_bank`.
- `response_confidence` = 1.0 for graded quizzes, 0.7 for AI-generated practice, 0.4 for ambiguous self-graded.

### 7. Continuous recalibration anchor set (+3–6% long-term)

Add a small curated "anchor questions" set per concept (5–10 stable items, marked `is_anchor=true` in `question_bank`). Nightly cron edge function `recalibrate-anchors`:

- Recomputes empirical `difficulty_b` from anchor responses only (stable population).
- Reshifts all non-anchor `difficulty_b` in the same concept by the mean anchor drift.
- Prevents long-term system-wide drift.

```sql
  ADD COLUMN is_anchor boolean NOT NULL DEFAULT false,
  ADD COLUMN confidence numeric NOT NULL DEFAULT 0.0;

ALTER TABLE graded_events
  ADD COLUMN concept_weight numeric,        -- weight at time of update

```

## Rollout order

1. **Migration** — schema + grants (no destructive change).
2. **Phase 1 (cheap wins, no infra):** Upgrades #1 (soft distribution) + #6 (gating) + #4 (coupling) — all in `ability-update` + `conceptInference.ts`.
3. **Phase 2:** Upgrade #5 (temporal decay) in `irtEngine.ts` + prompt fragment builder.
4. **Phase 3:** Upgrade #3 (dynamic b self-healing) in `ability-update`.
5. **Phase 4 (embeddings):** Upgrade #2 — generate centroids script, deploy `infer-concept` edge function, wire as fallback.
6. **Phase 5 (long-term):** Upgrade #7 — add `is_anchor` flag, deploy `recalibrate-anchors` nightly.

## Things this plan deliberately does NOT do

- **No model swap.** Stays on 1PL Rasch; 2PL/3PL discrimination is deferred (diminishing returns vs. complexity).
- **No realtime embedding on every answer.** Embeddings only run when keyword inference is weak, keeping cost flat.
- **No emotional/cognitive rewrite.** Those stay as flavor, never as theta input.
